<a name="gap-02"></a>
## GAP-02 · F1 `ChannelRouter.broadcast()` 缺失

| 属性 | 详情 |
|------|------|
| **优先级** | **P1** |
| **影响模块** | claw-channel（F1 消息渠道抽象） |
| **规范要求** | `ChannelRouter.broadcast(agents, msg)` — 向多个 Agent 广播同一条消息 |
| **现状** | `ChannelRouter`（[router.rs](../crates/claw-channel/src/router.rs)）只有 `route(msg) -> Option<String>`，采用 first-match-wins 语义，无广播方法 |

### 技术背景

`ChannelRouter` 内部用 `Arc<RwLock<Vec<RoutingRule>>>` 存储有序规则列表，`route()` 遍历后返回**第一个**匹配的 AgentId。这是严格的"一对一路由"语义。

规范在 F1 明确列出 `broadcast(agents, msg)` 作为独立方法，目的是支持：

- **监控 / 审计 Agent**：同时订阅所有渠道消息的副本，而不干扰主处理流
- **多 Agent 投票**：同一消息发给多个专家 Agent，由业务层聚合结果
- **渠道消息 fan-out**：一条消息同时触发多条并行处理管道

关键技术陷阱：广播语义与 first-match-wins 语义**互斥**，不能直接修改 `route()` 的返回类型（破坏 ABI），必须作为独立方法 `broadcast_route()` 提供。

当前的 `RoutingRule` 枚举包含：`ByChannel`、`BySenderId`、`ByPattern`、`Default`。实现广播需要引入"多匹配收集"模式，同时处理 `Default` 规则的语义（有显式匹配时，Default 不应追加以防重复投递）。

### 不修复的后果

- **监控 Agent 无内核机制**：实现"所有消息同时转发给监控 Agent"目前只能在应用层 hack（手动复制消息、两次调用 `steer()`），且每次 Agent 配置变更都要修改这段 hack。
- **A2A fan-out 需绕路**：多 Agent 协作起点通常是"向所有相关 Agent 广播任务描述"，缺失后应用层要自行维护 AgentId 列表并逐个发送，内核无法感知这层语义。
- **SDK 文档与实际不符**：Python/TypeScript SDK 用户按规范文档寻找 `channel.broadcast` IPC 方法，找不到。

### 修复方案

**方案 A（推荐）：在 `ChannelRouter` 上添加 `broadcast_route()` 方法**

```rust
// crates/claw-channel/src/router.rs

impl ChannelRouter {
    /// 返回所有匹配规则的 AgentId（多匹配语义）。
    /// 与 route() 的 first-match-wins 不同，此方法收集所有显式规则的命中结果。
    /// Default 规则仅在无显式匹配时生效（防止同一消息重复投递给 default agent）。
    pub fn broadcast_route(&self, msg: &ChannelMessage) -> Vec<String> {
        let rules = match self.rules.read() {
            Ok(r) => r,
            Err(_) => return vec![],
        };

        let mut agents: Vec<String> = vec![];
        let mut has_explicit_match = false;
        let mut default_agent: Option<String> = None;

        for rule in rules.iter() {
            match rule {
                RoutingRule::Default { agent_id } => {
                    default_agent = Some(agent_id.clone());
                }
                other => {
                    if let Some(agent_id) = other.matches(msg) {
                        if !agents.contains(agent_id) {
                            agents.push(agent_id.clone());
                            has_explicit_match = true;
                        }
                    }
                }
            }
        }

        if !has_explicit_match {
            if let Some(id) = default_agent {
                agents.push(id);
            }
        }

        agents
    }
}
```

同步在 IPC 层暴露：

```rust
// crates/claw-server/src/protocol.rs
#[derive(Deserialize)]
pub struct ChannelBroadcastParams {
    pub channel_id: String,
    pub content: String,
    pub metadata: Option<serde_json::Value>,
}

// crates/claw-server/src/handler.rs
"channel.broadcast" => handle_channel_broadcast(server, params).await,
```

**工作量估算**：router.rs 约 40 行，protocol.rs + handler.rs 约 50 行，测试约 60 行，**共约 150 行**。

## GAP-03 · F1 渠道 `send()` 路径无指数退避重试

| 属性 | 详情 |
|------|------|
| **优先级** | **P1** |
| **影响模块** | claw-channel（WebhookChannel、DiscordChannel） |
| **规范要求** | "发送失败自动重试（指数退避，最多 3 次），超出后报错给应用层" |
| **现状** | `WebhookChannel.send()` 和 `DiscordChannel.send()` 直接调用 reqwest，**无任何重试逻辑** |
| **代码状态** | `claw-pal/src/retry.rs`（155 行）已完整实现 `RetryConfig` + `with_retry_mapped`，**但未被任何 channel send 路径引用** |

### 技术背景

`claw-pal/src/retry.rs` 已提供完整的异步指数退避重试工具（全抖动算法）：

```rust
// 已存在，只是没有被使用：
pub struct RetryConfig {
    pub max_retries: u32,       // 默认 3
    pub base_delay: Duration,   // 默认 500ms
    pub max_delay: Duration,    // 默认 30s
}

pub async fn with_retry_mapped<F, Fut, T, E, M>(
    operation: F, config: &RetryConfig, is_retryable: M,
) -> Result<T, E>
```

当前 WebhookChannel 的出站路径（[webhook.rs](../crates/claw-channel/src/webhook.rs)）：
```rust
// 现状：直接发送，失败即返回错误
self.client.post(url).json(&msg).send().await?;
```

渠道发送失败的三类常见场景，全部应该透明重试：

| 场景 | HTTP 状态 / 错误 | 正确处理 |
|------|----------------|---------|
| Discord 全局限流 | 429 + `Retry-After` header | 等待 header 指定时间后重试 |
| Webhook 目标瞬时 5xx | 500/502/503 | 指数退避 1-2 次 |
| 网络层短暂丢包 | `reqwest::Error::Connect` | 立即重试 1-2 次 |

Discord 的特殊性：Discord API 有**两级限流**——全局（50 req/s）和路由级（`X-RateLimit-*` headers）。正确的重试必须解析 `X-RateLimit-Reset-After` header，使用 header 中的等待时间而非固定退避时间。

`claw-provider` 层的 LLM 请求已有完整的重试机制，渠道层没有复用，导致"LLM 调用失败会重试，消息发送失败不会重试"这一不一致性。

### 不修复的后果

- **消息静默丢失**：Discord Bot 遇到 429 时，消息被丢弃，用户无感知（应用层 catch 后通常只打日志）。生产环境下高峰期消息丢失率可达 5-15%。
- **Webhook 集成不稳定**：GitHub/Stripe 等外部系统 webhook 回调因目标服务短暂重启（5xx）导致触发失败，但实际上 1 秒后重试即可成功。
- **规范承诺失效**："发送失败自动重试"是内核层的显式承诺（kernel-features.md F1），目前实际上把重试责任转嫁给了每个应用层，碎片化地重新实现。
- **与 claw-provider 行为不对称**：LLM 调用失败会自动重试，消息发送失败不会，用户难以理解内核的一致性边界在哪里。

### 修复方案

**核心改动**：在 `claw-channel` 的 `Cargo.toml` 中添加 `claw-pal` 依赖，在 send 路径中使用已有的 `with_retry_mapped`。

```toml
# crates/claw-channel/Cargo.toml
[dependencies]
claw-pal = { path = "../claw-pal" }
```

```rust
// crates/claw-channel/src/webhook.rs
use claw_pal::retry::{RetryConfig, with_retry_mapped};

impl WebhookChannel {
    async fn send_with_retry(&self, url: &str, body: &serde_json::Value)
        -> Result<(), ChannelError>
    {
        let config = RetryConfig::default(); // max_retries=3, base=500ms
        with_retry_mapped(
            || async {
                self.client
                    .post(url)
                    .json(body)
                    .send()
                    .await
                    .map_err(|e| ChannelError::SendFailed(e.to_string()))
                    .and_then(|r| {
                        let status = r.status();
                        if status.is_server_error() {
                            Err(ChannelError::SendFailed(format!("HTTP {}", status)))
                        } else {
                            Ok(())
                        }
                    })
            },
            &config,
            |e| matches!(e, ChannelError::SendFailed(msg) if
                msg.contains("5") || msg.contains("Connect") || msg.contains("timeout")
            ),
        ).await
    }
}
```

Discord 专项（可作为独立 ticket）：
```rust
// crates/claw-channel/src/discord.rs
// 解析 X-RateLimit-Reset-After，覆盖默认退避时间
if let Some(reset_after) = response.headers()
    .get("X-RateLimit-Reset-After")
    .and_then(|v| v.to_str().ok())
    .and_then(|s| s.parse::<f64>().ok())
{
    tokio::time::sleep(Duration::from_secs_f64(reset_after)).await;
}
```

**工作量估算**：Cargo.toml 2 行，webhook.rs 约 30 行，discord.rs 约 30 行，测试约 50 行，**共约 112 行**。

## GAP-04 · F1 `UnifiedMessage` 缺少顶层 `sender_id` / `thread_id`

| 属性 | 详情 |
|------|------|
| **优先级** | **P2** |
| **影响模块** | claw-channel（ChannelMessage 结构体） |
| **规范要求** | `UnifiedMessage` 包含顶层字段：`channel_id, sender_id, content, thread_id?, metadata` |
| **现状** | `ChannelMessage`（[types.rs](../crates/claw-channel/src/types.rs)）：`id, channel_id, direction, platform, content, metadata, timestamp_ms`；`sender_id` 存入 `metadata["sender_id"]`，`thread_id` **完全缺失** |

### 技术背景

`sender_id` 存入 metadata 而非顶层字段，在以下场景产生实际问题：

**问题 1：`BySenderId` 路由规则存在运行时 panic 风险**

当前 ChannelRouter 的 BySenderId 匹配逻辑（[router.rs](../crates/claw-channel/src/router.rs)）：
```rust
// 运行时 JSON Value 查找 + 类型断言——编译器无法保证正确性
let meta_sender = msg.metadata.get("sender_id")
    .and_then(|v| v.as_str());  // 如果存的是数字类型则返回 None，静默路由失败
```

如果 channel adapter 实现者将 `sender_id` 存为数字（`metadata["sender_id"] = 12345`），路由规则会静默失效——没有编译错误，没有运行时 panic，只有消息被默认路由而不是精确路由。

**问题 2：`thread_id` 完全缺失导致多线程渠道路由失效**

Discord Thread、Slack Thread Reply、WhatsApp Group Reply 都需要 `thread_id` 来区分同一渠道的不同会话线程。没有 `thread_id`，所有来自同一渠道的消息都被视为同一个对话，导致：

- 同一 Discord 频道的不同 Thread 消息混入同一个 `HistoryManager`
- Agent 无法区分"回复 Thread A"还是"回复 Thread B"
- 多用户同时使用同一 Bot 时，对话历史互相污染

**问题 3：外部 SDK 接口形状不符**

规范承诺的 SDK 接口是 `msg.sender_id`，实际需要 `msg["metadata"]["sender_id"]`——使用 Python SDK 的开发者必须读源码才能知道正确的访问路径。

### 不修复的后果

- **Discord Thread / Slack Thread 场景下路由必然出错**：所有来自同一频道的消息合并到同一对话历史，群聊机器人无法正常使用。
- **BySenderId 路由规则脆弱**：任何新增 channel adapter 忘记在 metadata 中以正确类型写入 sender_id，路由就静默失败，调试极难（没有错误提示）。
- **SDK 文档不可信任**：`sender_id` 既然是规范中的一等公民，就必须能通过 `msg.sender_id` 访问，否则规范文档成为误导文档。

### 修复方案

此修复是**破坏性变更**（所有构造 ChannelMessage 的代码需更新），建议在 v1.6.0 随 semver bump 一起发布。

**方案 A（推荐）：添加顶层 `sender_id` 和 `thread_id` 字段，并提供 Builder 缓解迁移成本**

```rust
// crates/claw-channel/src/types.rs

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ChannelMessage {
    pub id: String,
    pub channel_id: ChannelId,
    pub direction: MessageDirection,
    pub platform: Platform,
    pub content: String,
    /// 发送方在该渠道的 ID（如 Discord user ID、WhatsApp phone number）
    #[serde(default)]
    pub sender_id: Option<String>,       // 新增顶层字段
    /// 会话/线程追踪（Discord Thread ID、Slack Thread TS、WhatsApp Group Reply ID）
    #[serde(default)]
    pub thread_id: Option<String>,       // 新增顶层字段
    pub metadata: serde_json::Value,
    pub timestamp_ms: u64,
}
```

同步更新 `BySenderId` 路由规则使用顶层字段：
```rust
// crates/claw-channel/src/router.rs
RoutingRule::BySenderId { sender_id, agent_id } => {
    // 直接访问顶层字段，无运行时类型断言
    if msg.sender_id.as_deref() == Some(sender_id) {
        Some(agent_id)
    } else {
        None
    }
}
```

`#[serde(default)]` 保证反序列化向后兼容——旧的 JSON 消息（无这两个字段）仍可正常解析为 `None`。

**工作量估算**：types.rs 约 10 行，router.rs 约 10 行，channel adapter 适配约 40 行（每个 adapter 构造消息处），测试更新约 40 行，**共约 100 行**。

<a name="gap-05"></a>
## GAP-05 · F1 inbound 消息→EventBus 链路未闭合

| 属性 | 详情 |
|------|------|
| **优先级** | **P2** |
| **影响模块** | claw-channel ↔ claw-runtime 集成（claw-server 接线层） |
| **规范要求** | "入站消息经过归一化后以统一格式送入 EventBus，由 AgentOrchestrator 分发" |
| **现状** | `RuntimeChannelEventPublisher` 已完整实现（[channel_event_publisher.rs](../crates/claw-server/src/channel_event_publisher.rs)，80 行，含 4 个测试），但 `KernelServer::new()` 中**没有构造该实例、没有将其注入到任何 channel adapter**，链路实际上旁路了 EventBus |

### 技术背景

`ChannelEventPublisher` trait（[claw-channel/src/traits.rs](../crates/claw-channel/src/traits.rs)）是专门为避免循环依赖设计的注入点——claw-channel 不依赖 claw-runtime，由更高层（claw-server）提供具体实现。

已完成的实现（`RuntimeChannelEventPublisher`）将 `ChannelEvent` 映射到 `Event`：

| ChannelEvent 变体 | 发布的 Event |
|------------------|-------------|
| `MessageReceived` | `Event::MessageReceived` |
| `MessageSent` | `Event::Custom("channel.message_sent")` |
| `ConnectionState` | `Event::Custom("channel.connection_state")` |

当前 `KernelServer::new()` 的实际初始化（[server.rs:354-439](../crates/claw-server/src/server.rs#L354)）：

```rust
pub fn new(config: ServerConfig) -> Self {
    let event_bus = EventBus::new();
    // ...
    let channel_router = Arc::new(ChannelRouterBuilder::new().build());
    // ❌ 缺失：没有构造 RuntimeChannelEventPublisher
    // ❌ 缺失：没有将 publisher 注入到 channel_registry 或任何 channel adapter
    Self { event_bus, channel_router, ... }
}
```

按规范应有的调用链：
```
channel.recv() → ChannelEventPublisher.publish() → EventBus → AgentOrchestrator 分发
```

实际的调用链：
```
channel.recv() → [IPC handler 手工处理，EventBus 从未收到渠道消息]
```

这条链路缺失的影响是系统性的，因为整个事件驱动架构的正确运转依赖 EventBus 收到渠道消息：

```
EventTrigger（订阅 EventBus）
    ↑ 永远等不到 ChannelEvent，因为 EventBus 从未收到
AgentOrchestrator（监听 EventBus 分发）
    ↑ 对渠道入站消息视而不见
AuditLog（记录消息接收）
    ↑ "渠道消息接收"事件从未写入
```

### 不修复的后果

- **EventTrigger 对渠道消息完全失明**：`trigger.add_event` 类型的触发器（监听 `message.received` 事件）永远不会被渠道消息触发，因为 EventBus 从未收到该事件。F6 的 EventTrigger 功能对渠道场景形同虚设。
- **AgentOrchestrator 无法基于渠道消息自动分发**：编排层对渠道输入视而不见，所有路由都必须由应用层手工实现，旁路了整个事件驱动架构。
- **审计日志缺少渠道消息记录**：AuditLog 没有"渠道消息接收"的条目，合规审计时无法追溯某条消息是否被处理。
- **架构意图失效**：EventBus 是内核的中枢神经，渠道消息旁路 EventBus 意味着整个"事件驱动"架构在渠道场景下是纸面架构，不是运行时现实。

### 修复方案

`RuntimeChannelEventPublisher` 已完整实现，修复只需在 `KernelServer::new()` 中添加**约 5 行接线代码**。

**方案 A（推荐，最小改动）：在 `KernelServer::new()` 创建并存储 publisher**

```rust
// crates/claw-server/src/server.rs — KernelServer::new() 中

let event_bus = EventBus::new();  // 已有
// ↓ 新增 3 行
let channel_event_publisher = crate::channel_event_publisher::RuntimeChannelEventPublisher::new(
    event_bus.clone(),
);

// KernelServer struct 新增字段：
// pub channel_event_publisher: Arc<dyn ChannelEventPublisher>,

Self {
    event_bus,
    channel_event_publisher,  // 新增
    // ...
}
```

然后在 `handle_channel_register`（handler.rs）中，注册 channel 时将 publisher 注入：
```rust
// 注册 channel adapter 时：
channel_adapter.set_event_publisher(
    Arc::clone(&server.channel_event_publisher)
);
```

**方案 B（全链路，同时完成渠道消息自动路由）**：

在 KernelServer 中启动一个后台任务，轮询 ChannelRegistry 中所有 active channel 的 `recv()`，经 ChannelRouter 路由后，通过 publisher 发布到 EventBus，再由 AgentOrchestrator 订阅并 steer 到目标 Agent：

```rust
tokio::spawn(async move {
    loop {
        for channel in channel_registry.list_active() {
            if let Some(msg) = channel.try_recv().await {
                let agent_ids = channel_router.broadcast_route(&msg);
                for agent_id in agent_ids {
                    orchestrator.steer(agent_id, SteerCommand::Custom(msg.content.clone())).await;
                }
                channel_event_publisher.publish(ChannelEvent::MessageReceived { ... }).await;
            }
        }
        tokio::time::sleep(Duration::from_millis(10)).await;
    }
});
```

**工作量估算（方案 A）**：server.rs 约 10 行，handler.rs 约 10 行，**共约 20 行**（不含已实现的 channel_event_publisher.rs）。

<a name="gap-06"></a>
## GAP-06 · F7 `AgentHandle` 不含 `resource_usage` 字段

| 属性 | 详情 |
|------|------|
| **优先级** | **P2** |
| **影响模块** | claw-runtime（agent_types.rs） |
| **规范要求** | `AgentHandle.resource_usage: CpuMs + MemoryBytes`（Agent 当前资源用量） |
| **现状** | `AgentHandle`（[agent_types.rs](../crates/claw-runtime/src/agent_types.rs)）只有 `agent_id: AgentId` + `event_bus: EventBus`；资源数据存在 `HealthStatus` 中（orchestrator.rs），但 `HealthStatus` 是独立结构，不是 `AgentHandle` 的一部分 |

### 技术背景

规范设计意图是让 `AgentHandle` 成为一个**当前快照**：调用方拿到 handle 就能知道该 Agent 的基本运行状态，无需另发请求。

当前资源数据的访问路径：
```
应用层  →  orchestrator.health_check(agent_id)  →  HealthStatus { cpu_usage_percent, memory_usage_kb }
```

问题：`HealthStatus` 是一次性轮询结构，不是 `AgentHandle` 的一部分。要获取资源数据，外部 SDK 需要额外一次 `agent.health` IPC 往返。

`AgentInfo`（[agent_types.rs](../crates/claw-runtime/src/agent_types.rs)）包含 `status`，但也不含 `resource_usage`。IPC 的 `agent.list` 响应目前缺少资源字段，与规范承诺的接口形状不符。

**平台限制说明**：
- **macOS / Windows**：精确的 CPU 时间需要不同的系统 API（`getrusage` / Windows Job Objects），实现复杂度高
- **tokio task 模式**：轻量 Agent 共享进程内存空间，无法精确隔离内存使用量
- **建议**：用 `Option<ResourceUsage>` 表达"有/无精确数据"，进程隔离的 Agent 提供精确值，task 模式 Agent 提供 `None`

### 不修复的后果

- **IPC `agent.list` 响应缺字段**：外部 SDK 按规范解析 `resource_usage` 时得到 `null`，需要额外调用 `agent.health` 才能获取资源信息，多一次往返。
- **多 Agent 资源调度困难**：编排层无法根据资源用量决定是否继续 spawn 新 Agent（需要单独轮询 health_check），无法实现基于资源的自适应扩缩容。
- **接口合约与文档不符**：文档驱动的 SDK 自动生成工具（如 OpenAPI/JSON Schema 生成）会产生与实际不一致的类型定义。

### 修复方案

**方案 A（推荐）：在 `AgentInfo` 中添加可选资源字段**

```rust
// crates/claw-runtime/src/agent_types.rs

/// Agent 当前资源用量快照。
/// 进程隔离 Agent 提供精确值；tokio task Agent 的字段为 None。
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub struct ResourceUsage {
    /// 累计 CPU 时间（毫秒）。tokio task 模式下为 None。
    pub cpu_ms: Option<u64>,
    /// 驻留内存估算（字节）。进程隔离模式下有效；task 模式为 None。
    pub memory_bytes: Option<u64>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct AgentInfo {
    pub config: AgentConfig,
    pub started_at: u64,
    pub process_handle: Option<ProcessHandle>,
    pub status: AgentStatus,
    pub resource_usage: Option<ResourceUsage>,   // 新增
}
```

在 `AgentOrchestrator` 的 `agent_info()` 中合并 `HealthStatus` 中已有的 CPU/内存数据填充此字段：

```rust
// orchestrator.rs 中 agent_info() 实现：
let resource_usage = self.health_status.get(&agent_id).map(|hs| ResourceUsage {
    cpu_ms: hs.cpu_usage_percent.map(|p| (p * 10.0) as u64),  // 估算，非精确
    memory_bytes: hs.memory_usage_kb.map(|kb| kb * 1024),
});
AgentInfo { ..., resource_usage }
```

**方案 B（最小改动）：IPC 层内联资源字段**

不改 Rust 类型，仅在 handler.rs 构造 `agent.list` JSON 响应时，合并 `HealthStatus` 字段：

```rust
// handler.rs 中 agent.list handler：
let mut info_json = serde_json::to_value(&info)?;
if let Some(health) = server.orchestrator.health_status(agent_id) {
    info_json["resource_usage"] = serde_json::json!({
        "cpu_ms": health.cpu_usage_percent,
        "memory_bytes": health.memory_usage_kb.map(|kb| kb * 1024),
    });
}
```

**工作量估算（方案 A）**：agent_types.rs 约 20 行，orchestrator.rs 约 15 行，**共约 35 行**。

<a name="gap-07"></a>
## GAP-07 · F7 轻量 Agent 以 tokio task 运行，崩溃可能传播

| 属性 | 详情 |
|------|------|
| **优先级** | **P2** |
| **影响模块** | claw-runtime（AgentOrchestrator） |
| **规范要求** | "每个 Agent 在独立进程（或隔离任务）中运行，**崩溃不传播**" |
| **现状** | 轻量 Agent 以 `tokio::spawn()` 运行；健康检查循环和 IPC 消息循环的 JoinHandle 如果没有正确处理 `JoinError::Panicked`，panic 状态的 Agent 可能被误认为"正常退出"而不触发 RestartPolicy |

### 技术背景

**tokio task panic 的实际行为**：

| 场景 | 结果 |
|------|------|
| `tokio::spawn` task panic，JoinHandle 被 await | `JoinHandle` 返回 `Err(JoinError::Panicked)`，panic 被捕获，进程不 abort |
| `tokio::spawn` task panic，JoinHandle 被 drop（detached） | panic 消息打印到 stderr，进程**不 abort**（tokio 4.x 默认行为） |
| `spawn_blocking` task panic | 同 spawn，被 JoinHandle 捕获 |
| mlua C 层触发 `abort()` | 整个进程**立即终止**，任何 Rust 层保护都无效 |

**当前实现的实际风险点**：

`AgentOrchestrator` 在 v1.4.1（G-10 修复）后，通过 `spawn_ipc_message_loop()` 管理 task，当 task 退出时触发 `trigger_restart()`。但有两个剩余风险：

1. **`spawn_ipc_message_loop` 中的 panic 路径**：如果 `task.await` 返回 `Err(JoinError::Panicked)`，当前的处理代码是否会区分"正常退出"和"panic 退出"并分别写入 AuditLog（`AgentStopped` vs `AgentFailed`），需要代码审查确认。

2. **mlua / C FFI abort 风险**：Lua 脚本 Agent 执行 C 扩展（如某些 mlua 绑定）时，如果 C 层触发 `abort()` 或 `SIGSEGV`，整个 claw-server 守护进程立即终止。这不是 tokio 的问题，是底层 C 不安全性，只有独立进程隔离才能解决。

**与 ProcessManager 模式的对比**：

| 模式 | 崩溃隔离 | 资源开销 | 适用场景 |
|------|---------|---------|---------|
| tokio task | ❌ C abort 无法隔离 | 极低 | 纯 Rust Agent |
| OS 独立进程（ProcessManager） | ✅ 完全隔离 | 高（进程启动开销） | 脚本 Agent、不信任代码 |

### 不修复的后果

- **脚本 Agent 崩溃传染**：Lua 脚本调用 C 库（如 luasocket、lfs 等 C 扩展）触发 segfault，会 kill 整个 claw-server 守护进程，导致所有 Agent 一起停止，而不是只停止问题脚本。
- **RestartPolicy 失效**：若 panic 在 JoinHandle drop 前未被正确检测（作为 Panicked 而非 Stopped），`trigger_restart()` 可能不触发，RestartPolicy 形同虚设。
- **审计日志漏记**：规范要求每次 Agent 启动/停止都写入 AuditLog。panic 退出如果走了"正常停止"路径，审计日志中不会有 `AgentFailed` 条目，导致合规审计出现"消失的 Agent"。
- **生产环境守护进程不稳定**：任何一个脚本工具 Agent 执行不受信任的 Lua 代码，都可能导致整个服务宕机。这对生产部署是不可接受的风险。

### 修复方案

**方案 A（推荐，短期）：在 task 包装层添加显式 `catch_unwind`**

```rust
// crates/claw-runtime/src/orchestrator.rs

use std::panic::AssertUnwindSafe;

// 包装 Agent task，捕获 panic 并触发正确的 RestartPolicy 流程：
let task_handle = tokio::spawn(async move {
    // AssertUnwindSafe 包装 — 我们保证不在 unwind 过程中访问 poisoned state
    let result = AssertUnwindSafe(async move {
        run_agent_loop(config, event_bus).await
    })
    .catch_unwind()
    .await;

    match result {
        Ok(Ok(_)) => {
            tracing::info!(agent_id = %id, "Agent stopped normally");
            update_status(&states, id, AgentStatus::Stopped);
            write_audit(AuditEvent::AgentStopped { agent_id: id });
        }
        Ok(Err(e)) => {
            tracing::error!(agent_id = %id, error = %e, "Agent exited with error");
            update_status(&states, id, AgentStatus::Crashed);
            write_audit(AuditEvent::AgentFailed { agent_id: id, reason: e.to_string() });
            trigger_restart(&states, id, &policy, &event_bus).await;
        }
        Err(panic_val) => {
            let msg = panic_val
                .downcast_ref::<&str>().map(|s| s.to_string())
                .or_else(|| panic_val.downcast_ref::<String>().cloned())
                .unwrap_or_else(|| "unknown panic".to_string());
            tracing::error!(agent_id = %id, panic = %msg, "Agent panicked");
            update_status(&states, id, AgentStatus::Crashed);
            write_audit(AuditEvent::AgentFailed { agent_id: id, reason: format!("panic: {}", msg) });
            trigger_restart(&states, id, &policy, &event_bus).await;
        }
    }
});
```

**方案 B（长期，彻底解决 C abort 问题）：脚本 Agent 强制走 ProcessManager**

对持有 `ScriptEngine` 的 Agent，spawn 时强制使用 `TokioProcessManager` 启动独立进程。需要为 ScriptAgent 提供独立的可执行入口（`claw-script-runner` bin），成本高但隔离最彻底。建议在 v2.0 规划中考虑。

**工作量估算（方案 A）**：orchestrator.rs 约 50 行修改，测试约 40 行（模拟 panic 场景），**共约 90 行**。

## GAP-08 · F6 Webhook 触发器 URL 格式与规范不符

| 属性 | 详情 |
|------|------|
| **优先级** | **P3** |
| **影响模块** | claw-server（trigger 注册逻辑）、claw-runtime（AxumWebhookServer） |
| **规范要求** | `POST /hooks/{trigger_id}` — 触发器 ID 直接映射到 URL 路径 |
| **现状** | `AxumWebhookServer` 接受任意路径；`TriggerAddWebhookParams.endpoint` 由调用方自由设定，`handler.rs` 不强制 `/hooks/{trigger_id}` 格式 |

### 技术背景

规范设计 `/hooks/{trigger_id}` 格式的两个意图：

1. **自描述性**：URL 本身包含足够信息，无需查数据库即可从日志 URL 推导出 trigger_id，极大降低排查成本。
2. **外部系统约定**：GitHub、Stripe、Slack 等外部系统配置 webhook 回调 URL 时，按约定构造 `/hooks/<id>` 即可，无需查配置文档。

当前实现允许调用方传入任意 endpoint（如 `"/github-push"`），注册后无法从 URL 反推 trigger_id：

```
日志：POST /github-push  →  需要查 TriggerStore 才能知道对应的 trigger_id 和目标 Agent
日志：POST /hooks/trg-abc123  →  trigger_id 直接可读，无需查表
```

此 gap 是**纯接口规范问题**，功能正常——webhook 照样能触发 Agent，只是接口语义不规范。

### 不修复的后果

- **监控/日志排查困难**：生产环境收到 webhook 回调日志时，无法直接从 URL 关联到对应的 trigger，排查需要额外查询 TriggerStore。
- **外部系统配置容易出错**：用户按文档写 `/hooks/{trigger_id}`，但实际服务端可能注册的是其他路径，不一致导致调试时间浪费。
- **文档信任度下降**：这类"文档说一套、实现是另一套"的细节积累多了，会侵蚀开发者对整个文档体系的信任。

### 修复方案

**方案 A（推荐，非破坏性）：`endpoint` 为空时自动使用 `/hooks/{trigger_id}`**

```rust
// crates/claw-server/src/handler.rs — handle_trigger_add_webhook()

// 修改前：直接使用 params.endpoint
let endpoint = params.endpoint.clone();

// 修改后：endpoint 为空或未提供时，自动规范化
let endpoint = params.endpoint
    .filter(|e| !e.is_empty())
    .unwrap_or_else(|| format!("/hooks/{}", params.trigger_id));
```

同时在 `protocol.rs` 中将 `endpoint` 改为 `Option<String>`：
```rust
// crates/claw-server/src/protocol.rs
pub struct TriggerAddWebhookParams {
    pub trigger_id: String,
    pub endpoint: Option<String>,   // None → 自动用 /hooks/{trigger_id}
    pub secret: Option<String>,
    pub target_agent: Option<String>,
}
```

**方案 B（破坏性，长期目标）**：完全移除 `endpoint` 字段，强制使用 `/hooks/{trigger_id}`。

建议：
- v1.5.0：实现方案 A，添加 deprecation warning 说明自定义 endpoint 在 v2.0 会移除
- v2.0.0：移除 `endpoint` 字段，强制规范路径

**工作量估算（方案 A）**：handler.rs 约 5 行，protocol.rs `endpoint` 字段类型改动 1 行，**共约 6 行**。
